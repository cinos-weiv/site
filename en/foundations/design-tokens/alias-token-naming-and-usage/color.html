Colors are categorized into two types: solid and translucent.

  • Solid colors: Prefixed with vsds-sys-color-, used for all semantically meaningful colors such as primary, background, text, and status colors.
  • Translucent colors: Prefixed with vsds-sys-color-opacity-, used to define opacity variations on semantic colors like neutral-sm or primary-md.

This classification allows Color tokens to be applied more flexibly across various UI scenarios while maintaining a consistent naming logic and scalability.

 

 

Solid colors

Solid colors are classified into four categories based on their intended use and design semantics: Semantic Colors, Surface Colors, Text Colors, and Functional-only Tokens. This structure helps clarify usage rules and pairing principles, ensuring visual consistency and accessibility quality. The following sections will introduce the design goals, usage scenarios, and pairing suggestions for each category.

 

Semantic Colors

Semantic colors are the primary color schemes used to express product personality and interface status.

Main usage
  • primary / secondary / tertiary: These represent the brand identity and visual signature of the product. They are commonly used for key interactive elements, navigation, and emphasized functional areas.
  • neutral: Neutral tones are applied to emotionally neutral elements, such as field borders, button dividers, or other visually neutral components.
  • info / success / warning / danger: These are situational colors used to emphasize feedback states (e.g., hints, success messages, warnings, and errors). They can be combined with primary color sets to maintain a coherent interface tone.
Use of variants
  • These categories can all be extended using the *-variant pattern (e.g., primary-variant, info-variant). Variant colors are used for interactive states (like hover or active), visual hierarchy, or high-contrast scenarios, offering greater visual diversity and flexibility.
Usage in content
  • These semantic colors can be applied to the background of interactive components (e.g., Button, Toast, Badge), and can also be paired with on-* variants for foreground elements like text or icons.
Requirement
  • Since semantic colors form the foundation of system component design, they are Required and must not be omitted or left undefined. Make sure every theme has a complete definition to maintain design language consistency and ensure accurate component rendering.

Surface Colors

Surface colors are used for page or module backgrounds, and may also be applied to the body of components. However, they are not intended for use on text or icons. Beyond setting background color, surface colors also define the visual hierarchy and spatial depth of the interface, working closely with shadows and layering.

To maintain consistent pairing, colors from different series must not be mixed (e.g., surface-100 must only be paired with text-100).
VSDS provides four types of surface color sets, named according to light/dark tones and semantic contexts, to support background needs across various scenarios.

Main usage
  • surface-100 / surface-200: Represent background colors at different visual levels. The higher the number, the more elevated or spatially prominent it appears.
  • Typically used for primary layout backgrounds (e.g., cards, content sections, foundational modules), and should be paired with text-100 to text-200 to ensure readability.
Use of variants
  • surface-variant-100 / surface-variant-200: Used for stylistic variations within the same theme, such as secondary modules or sections. These help emphasize visual hierarchy while maintaining overall color harmony.
  • Also suitable for interactive states such as hover or selected areas.
Contrast usage
  • In practical design scenarios, regardless of whether the default theme is light or dark, there are often cases where strong contrast is needed—such as for overlay components, navigation bars, or card modules. In such situations, surface-inverse-* can be used to create backgrounds in contrast to the main theme palette, helping to emphasize hierarchy and focal points.
  • When using inverse surface colors, be sure to pair them with the corresponding text-inverse-* to ensure sufficient contrast and readability on dark surfaces.
Background usage only
  • Surface colors are intended for background use only. They must not be used for foreground elements such as text, icons, or borders. Common applications include component bodies, foundational layout blocks, and system interface backgrounds.
Requirement
  • To ensure all components can correctly match styles and background layers, surface-100 / 200, surface-variant-100 / 200, surface-inverse-100, and surface-inverse-variant-100 must be defined for every theme. These are the foundational background colors in the design language and must be fully configured.
  • Other layers, such as surface-300+, or extensions starting from surface-variant-200 or surface-inverse-200, can be added flexibly depending on product needs (see surface color overview table below).
  • Note': that all surface colors are for background use only and must be paired with the corresponding text-* series. Mixing incompatible series may result in inconsistent color logic or reduced readability. This approach helps preserve visual consistency and information clarity across the interface.

Text Colors

Specifically designed for text and icons, and must be paired with the appropriate surface background.

Main usage
  • text-100 / 200 are primarily used for text and icons paired with surface-100 / 200. Typically found in main content areas and key interface elements. A lower number indicates a higher visual hierarchy and stronger contrast (e.g., headings), while higher numbers are suited for secondary or supportive content.
Use of variants
  • Variant types are available for the above text tokens to support visual hierarchy across different contexts (e.g., secondary text, helper icons). For example, text-variant-100 can be used for hint text or de-emphasized content, helping to clarify the structure of the layout.
Contrast usage
  • The text-inverse-* series is designed to be used with surface-inverse-* background areas. Even in a predominantly light-themed interface, there may be darker module areas such as overlays or navigation bars. In such cases, text-inverse-* should be used. This contrast pairing effectively separates visual layers while maintaining consistent visual logic and readability.
Usage in content
  • All text-* tokens are intended only for content elements such as text and icons. They must not be used for component backgrounds or solid color blocks.
  • If the background belongs to a semantic color group (e.g., primary, secondary), use the corresponding on-* tokens (e.g., on-primary, on-secondary) to ensure content elements have sufficient visual contrast and legibility.
Requirement
  • text-100 and text-200 are required definitions for every theme. They ensure proper representation of primary and secondary content text. Additional levels like text-300 and variant types can be extended as needed.
  • Similar to surface tokens, text-inverse-* and text-inverse-variant-* offer flexibility for advanced contrast scenarios but are not mandatory. Adjust their use according to product needs and always maintain consistent contrast logic and hierarchy.

The system defaults to providing surface-100 ~ surface-300 as the primary hierarchy levels. It is recommended to keep each set of surface-* and text-* tokens within 3 to 4 levels to maintain consistency in styling and clarity in semantic meaning.
An excessive number of token levels may make it harder to distinguish their purposes in design, reducing semantic readability. It also increases the burden on engineering teams during maintenance and theme switching, potentially impacting WCAG contrast checks and ultimately decreasing overall usability and stability of the interface.

 

Functional-only Tokens

Functional-only tokens are designed for specific use cases only. They must not be mixed with or used as general semantic colors (such as primary or neutral), to avoid semantic ambiguity and visual inconsistency.

  • link / link-variant: Used exclusively for clickable text links. link is the default style, while link-variant is used for interaction states (e.g., hover). Must not be applied to non-link text.
  • disabled / disabled-variant / on-disabled :Used to indicate disabled states for components or text. disabled is the default style, disabled-variant adds visual depth (e.g., darker tones without interactivity), and on-disabled is used for foreground elements (text or icons) in the disabled state.
  • outline / outline-variant / outline-inverse / outline-inverse-variant: Used for component border colors. outline is the standard border for interactive elements (e.g., fields, buttons) and must meet WCAG AA contrast requirements (recommended ratio: 3:1). outline-variant is for non-interactive, decorative elements (e.g., dividers, module outlines). The inverse series is intended for use on contrasting (dark) backgrounds.
  • focus-outline: Used solely for keyboard focus indicators. Should be applied to focusable components like fields and buttons, and must meet visibility and contrast guidelines (WCAG 2.1 AA, recommended contrast ratio: 3:1 or higher).
Requirement
  • Functional-only tokens are not for decorative purposes—they are essential design tools for reinforcing interaction semantics and state recognition. These tokens are tightly coupled with component behavior. Ensure that every theme includes a complete definition of all functional-only tokens, and apply them strictly according to their intended use. Do not mix them with general semantic colors.

 

 

Translucent Color Group

Translucent color tokens are prefixed with vsds-sys-color-opacity-, used to define opacity tiers based on specific semantic colors (e.g., primary, neutral). These tokens can be applied in various scenarios involving visual layering control, such as backgrounds, overlays, and button layers.

The system currently supports four semantic color groups—primary, secondary, neutral, and neutral-inverse. Each group includes a series of opacity sizes (e.g., sm, md, lg), allowing for varying degrees of transparency such as overlays.
