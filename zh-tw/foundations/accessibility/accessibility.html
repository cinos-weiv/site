Accessibility

讓產品從設計階段開始具備無障礙性，先理解使用者的實際情境與需求

無障礙設計是提升產品品質與使用公平性的設計策略，做得好時，使用者不一定會察覺產品「特別幫了誰」，但會自然感受到這是一個好用、友善、誰都能使用的設計。
 

從 Accessibility 出發

在實際的教育與行政環境中，學生、教師、學校管理層人員、設備管理人員，各種角色都可能遇到使用限制。像是閱讀困難、暫時性的手部受傷、年長導致的聽力退化，甚至只是環境光線太強、裝置反光等，都會影響使用者操作產品的體驗。
舉例來說：

  • 學生可能在使用數位教材時遇到閱讀障礙或需依賴鍵盤操作的輔助設備。
  • 教師在講課時需快速切換介面、書寫、點選操作，若回饋不明確或按鈕太小，會造成操作壓力。
  • 管理層人員需審視整體系統是否公平提供每位使用者平等的參與機會。
  • 設備管理者則要確保裝置與平台能支援各種輔助技術，且可穩定維護。

此外，隨著產品拓展至商務場域，無障礙同樣的扮演關鍵角色。
無論是遠端協作、會議系統、展示設備，若未納入使用者限制與操作彈性，將降低整體體驗與被採用率。
ViewSonic Design System 也將無障礙視為核心價值之一，並持續整合於元件樣式、Token 設定與交付文件中。
即使目前應用多數來自教育場域，我們將開始針對商用與協作設備預設納入無障礙設計，並持續擴展相關應用，確保每位使用者都能流暢的使用產品。
無障礙設計不是為了特定少數人，而是為了確保所有人在任何狀況下都能順利使用我們的產品。 這不僅是法規上的要求，更是實際提升產品品質與使用體驗的基本設計責任。
統計指出：

  • 全球約有 3 億至 3.5 億人受到色覺障礙影響，約佔人口 4.5%，其中男性約佔 8%。
  • 約有 10–20% 的人口具有語言型學習障礙（如閱讀障礙），在教育環境中每十人就可能有一人受到影響。
  • 全球約有 4.3 億人有中度以上的聽力障礙，其中包含 3,400 萬名兒童。
  • 至少 22 億人有視覺障礙問題，其中 超過 10 億人尚未獲得妥善治療。
完整資料請參考下方資料來源連結。

 

什麼是 Accessibility？

Accessibility 是一套設計原則，目的是讓所有人無論能力、年齡或環境條件都能有效存取、理解並操作數位內容。

常見無障礙設計面向：

  • 視覺面向：色彩對比、替代文字、字體清晰、支援螢幕閱讀器
  • 聽覺面向：字幕、文字稿、視覺提示
  • 認知面向：語言簡潔、結構清晰、降低干擾
  • 操作介面：鍵盤操作支援、欄位正確標示、合理焦點順序

 

從設計開始落實 Accessibility

不需要一次就達到全部的標準，而是從設計階段便開始思考使用的限制與情境，便能實質提升產品的可接近性。

起手點可以包含以下自我檢查項目：

  • 文字是否清楚？顏色對比是否足夠？
  • 圖片是否有提供替代文字？
  • 所有操作是否支援鍵盤或觸控？
  • 使用者是否能理解操作流程與錯誤訊息？
更多完整檢查項目，請參見下一章節的 Accessibility Checklist。

 

我們對齊的 Accessibility 規範

教育產品在設計與開發階段，更應納入特定無障礙功能，以符合美國法律要求，並確保所有學生，特別是身心障礙學生，能完整參與學習流程。這些功能不是建議事項，而是受以下法規明文規定：

  • ADA（Americans with Disabilities Act）：要求數位內容與工具對身心障礙者可使用，確保平等使用資源。
  • Section 504（康復法第 504 條）：所有接受聯邦補助的學校須提供合理調整與協助。
  • IDEA（身心障礙者教育法）：保障身心障礙學生在最少限制環境中獲得適當教育。
  • WCAG（Web Content Accessibility Guidelines）：針對網站與應用程式內容的無障礙標準，通常以 WCAG 2.1 AA 等級 為最低門檻。

標準

實務對應

WCAG A+AA

保證足夠對比、可操作性、明確的語意與結構、避免只靠色彩傳達資訊等

ADA／504／IDEA

特別適用於教育平台、公部門與接受政府補助之產品或系統

若產品應用於教育現場或公共服務平台，無障礙設計屬於法律規範項目，必須依據相關法規進行實作與驗證。

 

Accessibility design 的驗證方式

在設計與開發流程中，不需死背所有無障礙規範，但應熟悉基本檢查項目與測試方法，並預留驗證的空間。

建議導入的檢查流程：

Design system 層級

適用於規劃／維護設計系統時：

  • 確保所有 Color Token 顏色皆符合文字與背景的對比度建議。
  • 文字大小應符合最佳閱讀性，建議 Body text 不小於 16px（或依照平台設備調整）。
  • 確保 Components 套用的 Alias Token 經過色彩與文字對比測試。
  • 所有互動元件（如 Button、Checkbox）需具備足夠的點擊或觸控範圍，無特殊需求建議至少 44px × 44px。
  • 每個元件都應具備 Focus 樣式，其外框對比度建議不低於 3:1，方便鍵盤操作或高對比需求者辨識。
Tips：尺寸與單位建議
所有與尺寸相關的數值（如 16px、44px）皆以 CSS 像素（CSS px）為單位。
若是在 iOS 或 Android 上開發，請轉換為 pt 或 dp。

 

UI Mockup 層級

適用於設計稿（Mockup）交付前的畫面檢查：

  • 確保元件排版與行距、間距清晰、易於閱讀。
  • 資訊呈現不應僅依賴顏色或圖示（如紅色提示或感嘆號符號），應搭配文字說明。
  • 錯誤提示需 明確指出錯誤原因 與 如何修正（例：「請輸入有效的電子郵件」）。

 

無障礙驗證支援文件

適用於準備驗證、合規或與開發團隊交付同步時：

  • 提供 Focus 操作順序 的註記標示（對應 WCAG 的「可操作性」原則，建議順序符合螢幕閱讀器自然邏輯）。
  • 提供 Screen Reader Labels 建議內容，標示哪些元件應有 aria-label 或輔助說明文字。

 

常見檢查工具：

  • Stark（Figma Plugin）: 檢查設計稿中的色彩對比與文字可讀性。
  • A11y - Focus Order（Figma Plugin）: 針對互動流程進行焦點順序標註與檢查。
  • VoiceOver（macOS）／NVDA（Windows）：模擬螢幕閱讀器的實際操作體驗，推薦搭配無障礙標註一同測試。
  • Axe ／ WAVE ／ Lighthouse: 針對開發後的頁面進行整體檢測，包含結構語意、可操作性與色彩對比等。
  • VPAT 模板: 協助對照並記錄是否符合常見無障礙規範項目（尤其適用於大型專案或交付文件）。
